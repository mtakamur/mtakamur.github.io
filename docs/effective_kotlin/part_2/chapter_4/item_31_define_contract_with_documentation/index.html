Item 31: Define contract with documentation

更新日: 2022/08/20

まとめ

  • 本節における “Contract” とは、クラスやメソッドの実装者と利用者側の共通認識となる、メソッドの動作、利用方法やクラスのパラメータの意味のこと
  • ドキュメンテーションによって、API の利用方法、動作、パラメータなどについての説明を書こう
  • ただし、むやみやたらと細かく書けばよいわけではない
    • 名前から意味が明らかなメソッドやパラメータに対してドキュメンテーションがあっても追加で得られる情報がないので無価値
    • 書いたこと(約束)は守らねばならない。逆に言えば、ドキュメンテーションに書いていないことはある程度自由に変更できる。曖昧さは極力排除すべきではあるが、極端に制約を設けるとメンテナンスがしづらくなる
  • 以下のものは Contract として有効
    • クラス、メソッド、パラメータの名前
    • クラス、メソッド、パラメータのドキュメンテーションや、実装中に記載されたコメント
    • ジェネリクス制約や戻り値の型などの型の情報

ちょっと詳しく

ある API を開発して、外部に公開するようなシチュエーションを考えます。

クラスやメソッドの命名だけでは、API の動作や要求するパラメータについて一切の曖昧さなしに語りつくすことはできないでしょう。 このような状況に出くわしたとき、API の利用者側は API の動作や要求しているパラメータが何であるかを探るために実装を見る必要が出てきます。 もしソースを公開していないとしたら、利用者側としてはお手上げです。

API の動作・利用方法や、メソッドが要求するパラメータや戻り値についての情報はドキュメンテーションとして利用者側に公開されているべきです。 Kotlin では、KDoc によってドキュメンテーションを記載することができます。 KDoc の記載の仕方については公式サイトに譲ることとして、この節では何を記載すればクリアな API になるかを考えます。

Contract(実装者と利用者の間の共通認識)

この節では Contract という言葉が頻繁に登場しますが、直訳の"契約"という意味だとしっくりきません。 いまいち上手な解釈ができていませんが、ここでは “API の実装者と利用者の間に築かれるクラスやメソッドの動作に関する共通認識” と考えると誤解がないと思います。

API を開発、利用するにあたって、contract が適度に記載されていることは実装者と利用者双方にとってメリットがあります。

まず利用者としては、contract として記載されていることは必ず満たされるものとして扱うことができます。 実装を全部見て、利用方法や動作を確認する必要はなく、contract を読めば、それらについては理解することができます(適度に contract が記載されている場合に限ります)。

実装者としては、変更してよい領域とダメな領域の境界が明確となり、API の将来にわたるメンテナンスが容易になります。 API の動作や要求は contract の記載を満たす必要がありますが、逆に言えば、記載のないことについて変更するのはルール違反ではないということになります。

例えば、ユーザーに対してメッセージを表示するための次の API を考えます。

/**
* Universal way for the project to display a short
* message to a user.
* @param message The text that should be shown to
* the user
* @param length How long to display the message.
*/
fun Context.showMessage(
  message: String,
  duration: MessageLength = MessageLength.LONG
) {
  val toastDuration = when(duration) {
  SHORT -> Length.SHORT
  LONG -> Length.LONG
}

この showMessageというメソッドには、このメソッドはユーザー対して何らかのメッセージを表示するものであるという contract が記載されています。 メッセージの表示の仕方については具体的な記載がありませんので、実際にどのような見た目でメッセージを表示するかは実装者に委ねられており、いつでも変更可能なものです。 Android での実装を例とすれば、メッセージを Toast として表示するか SnackBar として表示するか、もしくは他にオリジナルで実装するかは実装者が好きに選んで問題ないことになります。

曖昧さをなくすことが contract の役割ではありますが、contract を詳細に書くたびに実装者は守る必要のある制約を増やしていることになりますので、contract の詳細度は状況に応じて適切なレベルにとどめておくのがよさそうです。

Defining a contract(contract の定義)

ドキュメンテーションによって contract が記載されると述べましたが、ドキュメンテーションのみが contract となりうるわけではありません。 以下のようなものも立派に contract としての役割を果たすことができます。

  • クラス、メソッドやパラメータの名前
  • コード中のコメントやドキュメンテーション
  • パラメータとして要求する値やメソッドの戻り値の型の情報

例えば次のようなメソッドを考えます。

/**
* Increments [value] by one.
*/
fun incrementByOne(value: Int) : Int { // ...(1)
  return value++
}

メソッド名、incrementByOne が十分に contract として機能しています。 このメソッドの動作として、特に他に記載すべきこともないでしょう。

このような状況では、上記のような KDoc によるドキュメンテーションはむしろ書かないほうが良いです。 KDoc の記載によって、メソッド名からわかることに追加して得られる情報がないからです。

また、パラメータとして要求する値やメソッドの戻り値の型の情報も大切な contract の一部になり得ます。 (1)の例では、型推論が可能なので、メソッドの戻り値が Int であることを省いて次のように書くこともできます。

fun incrementByOne(value: Int) = value++

この程度のメソッドであれば、戻り値の型の記載が省略されても容易に把握できますが、Int 型であることが明記してある場合に比較すれば、返される値の型が見づらくなっています。 型推論による記述は、コード自体をシンプルに保つことができる反面、コードの読み手にも型の推論を強制することとなるため、あまり複雑なメソッドで利用するとコードが読みづらくなります。

Type system and expectations

クラスの継承やインターフェースの実装時の注意点です。

あるクラスを継承してサブクラスを実装するとき、親クラスのメソッドをオーバーライドできることがあります。 サブクラスでオーバーライドするメソッドは、親クラスで定義されている contract を守る必要があります。 なぜなら、サブクラスのインスタンスは親クラスのインスタンスとして用いられることがあるからです。 「ペットを飼っている」と口では言っても、「犬(などの動物)を飼っている」というのが実態でしょう。

サブクラスはあくまで親クラスを拡張した、より具体的なクラスであるべきですので、親クラスの持つメソッドをまったく別のものにオーバーライドすることはやるべきではありません。 このことは一般に Liskov substitution principle (リスコフの置換原則)として知られています。 この原則は、“S が T を継承したサブクラスであるとき、S は常に T として扱うことができる” ということを意味します。 単純な例として、犬は常にペットとして扱うことができます。(その逆は時としてダメ。)

もしも、サブクラスでのオーバーライドが親クラスのメソッドを一切踏襲していないとしたら、上記の原則は成り立たないことになってしまいます。 あるクラスを継承したクラスを実装する際には、親クラスが満たしている contract を確認し、実装するサブクラスでもその contract を満たすように設計する必要があります。
