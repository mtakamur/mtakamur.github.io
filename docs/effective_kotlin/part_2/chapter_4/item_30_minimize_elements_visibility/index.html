Item 30: Minimize elements visibility(メソッドや変数はできるだけ見えないように)

更新日: 2022/08/13

まとめ

  • 外部に公開するメソッドや変数はできるだけ減らそう
  • 見える必要のないものは見えないほうが幸せ。知る必要のないことは秘匿されていれば、利用に際して学習が必要な部分が限定できる。
  • 特にインスタンスの状態を表すような変数は絶対に外部から変更できるようにしてはならない。逆に、外部から変更できるような変数はインスタンスの状態を表すための変数として不適切。
  • private なメソッドや変数はいつでも public にできるけれども、その逆はなかなか困難。すでに誰かが外部から参照しているものは好き勝手に private に変更するわけにはいかない。

ちょっと詳しく

主にクラスのメソッドや変数の可視性の適切な選び方についての話です。 Kotlin に限らず、オブジェクト志向言語については共通認識として考えてよいです。

基本スタンスは、外部に公開するメソッドや変数は必要最低限にすることです。

クラスの実装者と利用者それぞれの立場から

クラスの利用者の立場としては、public なメソッドや変数が少ないほうが学習コストが低くなります。 利用に際して読み込むドキュメンテーション(や、時にはソースコード本体)はできるだけ少ない方が良いでしょう。 利用者が参照する必要のないメソッドや変数まで visible になっていると、本来必要のないドキュメンテーションやソースコードまで読むことになり、負担になります。

利用者側に優しいのはもちろんのこと、実装者側にもメリットがあります。 それは、クラスを変更しやすくなることです。 極端なことをいえば、invisible なメソッドや変数はいつでも勝手に変更しても問題ありません。 なぜなら外から見えないからです。 利用者側としては visible なメソッドや変数がドキュメンテーション通りの動作をしてくれればよいわけですから、その内部の処理として何がなされていようと基本的に興味はないわけです。

変更のしやすさという側面からはもう一点、invisible なものはいつでも visible にできますが、その逆は容易ではありません。 すでに誰かが利用しているものを実装者が勝手に invisible に変更すれば、きっとクレームをいただくことでしょう。 利用者側からすれば、これまで使えていたものが突然使えなくなることを意味します。

プロパティの可視性

クラスのプロパティは、しばしばインスタンスの状態を表すために用いられます。 もしそのプロパティが、外部から変更可能に実装されている場合、そのプロパティはもはやインスタンスの状態を意味しません。 インスタンスの状態は、あくまでもインスタンスの事情によって設定されるべきで、外部から任意に書き換えられてはなりません。

ただし、immutable なプロパティについては外部に公開されていても問題ない場合もあります。 理由は簡単で、変更不可能だからです。

特に、Kotlin における data class のプロパティは、基本的にすべて public であるべきといえます。 データを保持して公開することが data class の目的のはずだからです。

data class のプロパティを 下記のように public val として定義すればそのインスタンスは状態のセットとして用いることができます。

class User(
  val name: String,
  val age: Int,
)

上記の data class のプロパティはいずれも immutable な変数として定義していますので、外部から変更される恐れはないように見えるのですが、Kotlin の便利なようで厄介な copy メソッドによってあたかも name や age が mutable であるように見える処理も書けてしまうので、その点は注意が必要です。

main() {
  val user1 = User(name = "hogehoge", age = 28)
  val user2 = user1.copy(name = "fugafuga")
}

もちろん、上記 user1 のプロパティが直接変更されているわけではないのですが、user2 のプロパティは user1 の name プロパティだけを任意に書き換えたものと等価です。 copy メソッドによって、コピー元のプロパティをすべて引き継ぎつつ、任意のプロパティを任意の値で書き換えたインスタンスが作れます。

Using visibility modifiers(可視性修飾子)

Kotlin では、class 内のメソッドや変数については以下の 4 種類の可視性を設定することができます。

  • public: デフォルトの可視性で、どこからでも見える
  • private: 自分自身からしか見えない
  • protected: 自分自身、もしくはこのクラスを継承するクラスからしか見えない
  • internal: 同じモジュール内からしか見えない

Kotlin におけるモジュールとは、コンパイル単位に相当するもので、Gradle によってビルドしている環境では同一ファイル内となります。

トップレベルのメソッドや変数では、上記のうち protected を除いた modifier が利用可能です(トップレベルメソッドや変数には継承の概念がないため protected の適用はない)。
